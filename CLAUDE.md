# Problem Solving Gym — CLAUDE.md

Astro static-site scoreboard for an internal coding-problem gym. Problems
live on disk under `problems/<slug>/`; each developer commits a solution +
a `results.md` (generated by the problem's Go-compiled evaluator binary);
the site reads all of that at build time and renders ranked leaderboards.

## Repo layout

```
src/
├── layouts/Layout.astro       # global tokens, fonts, dark theme
├── lib/data.ts                # getAllProblems(), getProblemWithScoreboard(slug)
└── pages/
    ├── index.astro            # homepage — problem cards (newest first)
    └── problems/[slug].astro  # dynamic route — markdown + leaderboard

problems/
└── <slug>/
    ├── problem.md             # title/slug/published_at frontmatter + body
    ├── evaluate               # Go binary, macOS arm64
    ├── evaluate-linux-amd64   # Go binary, Linux
    ├── evaluate.exe           # Go binary, Windows
    └── submissions/<handle>/  # solution.* + results.md

tools/                          # shared verifier binaries
└── verify, verify-linux-amd64, verify.exe

.github/workflows/pr-checks.yml # CI: HMAC integrity validation on PR

_eval-src/                      # GITIGNORED — Go source for evaluators + verifier
```

## Local dev

npm install

## Your first submission in 5 minutes (participant tutorial)

Pick **1 and Prime Numbers** for this walkthrough — it's `problem-1`..

### 1. Clone and boot the site

```
git clone https://github.com/Axy-Science/Axy-AI-Gym.git
cd Axy-AI-Gym
npm install
npm run dev
```

Open `http://localhost:4321/` and click **1 and Prime Numbers**.

### 2. Read the problem

Read `problems/problem-1/problem.md`. The contract: integer N on stdin,
print qualifying numbers less than N one per line. **Read the example.**

### 3. Make your submission folder

Use your GitHub handle as the folder name. Examples here use `aarsh`.

```
mkdir -p problems/problem-1/submissions/aarsh
```

### 4. Write your solution (any language)

Anything that reads from stdin and writes to stdout. Quick Python stub at
`problems/problem-1/submissions/aarsh/solution.py`:

```python
import sys
n = int(input())
# your function goes here
```

Node, Go, Rust, anything — all fine.

### 5. Run the eval

From inside the problem folder:

```
cd problems/problem-1
# macOS (Apple Silicon)
./evaluate "python3 submissions/aarsh/solution.py" submissions/aarsh/results.md
# Linux
./evaluate-linux-amd64 "python3 submissions/aarsh/solution.py" submissions/aarsh/results.md
# Windows
.\evaluate.exe "python submissions\aarsh\solution.py" submissions\aarsh\results.md
```

You'll get a per-test breakdown and an overall score out of 100. The eval
runs 10 test cases — 5 correctness, 5 efficiency. Tier 2 has strict
timeouts; brute-force algorithms will TIMEOUT on the big inputs.

### 6. Iterate

If a test fails, the generated `results.md` shows expected vs. got for
each failing case. Read it, fix the bug, re-run — it overwrites the
previous `results.md` with a fresh timestamp + checksum.

The dev server (`npm run dev`) auto-refreshes when `results.md` changes,
so you can watch your row update live at
`http://localhost:4321/problems/problem-1`.

### 7. Commit and open a PR

```
git checkout -b aarsh/problem-1
git add problems/problem-1/submissions/aarsh
git commit -m "solve: problem-1 (aarsh, 100/100)"
git push origin aarsh/problem-1
gh pr create --fill
```

PR CI runs `tools/verify-linux-amd64` against your `results.md` and rejects
the PR if the HMAC checksum doesn't match the recorded score — i.e. don't
hand-edit the file. Once your PR merges, the deployed site rebuilds and
your row appears.

### Scoring & tiebreakers

- Ranks sort by **score descending**.
- Ties break by **submission timestamp ascending** — earliest wins.

## Adding a new problem (maintainers)

See [`MAINTAINER.md`](./MAINTAINER.md) for the full guide. **All new
problems must use the Go binary + HMAC integrity evaluator** (see
`problem-1` as the reference). Source lives under `_eval-src/`, which is
gitignored. `problem-2` predates this rule and uses a bash evaluator —
do not copy its layout for new work.

## Notes for AI assistants working in this repo

- **Data is filesystem-driven.** `src/lib/data.ts` reads `problems/` at build
  time via `process.cwd()` + `fast-glob`. No DB, no API. Touching files
  invalidates the dev build automatically.
- **Markdown rendering:** problem body → `marked.parse()` → `<article set:html>`.
  Markdown's first `<h1>` is hidden via CSS since the page already shows a
  styled title.
- **Dynamic routes:** `[slug].astro` needs `getStaticPaths()` returning every
  problem slug; without it Astro can't pre-render those pages.
- **Design tokens:** all colors, fonts, spacing live as CSS variables on
  `:root` inside `src/layouts/Layout.astro`. Edit there, not per-page.
- **Fonts:** `Press Start 2P` is marquee-only (unreadable at body sizes);
  `JetBrains Mono` is everything else.
- **Ranking tiebreaker** is dense — entries with identical `(score,
  submitted_at)` share a rank. Implemented in `getProblemWithScoreboard`.
- **macOS case quirk:** the filesystem is case-insensitive but git is
  case-sensitive. Don't reintroduce a lowercase `readme.md` alongside
  `README.md`.
- **Never commit sample/reference solutions.** Problem-1's reference
  Python sieve and the suboptimal alice/bob solutions live in
  `_eval-src/` (gitignored) and are used only for evaluator calibration
  and local end-to-end testing. The scoreboard ships empty so
  participants can't crib from a committed reference.
- **HMAC secret** is compiled into the verifier and evaluator binaries
  via `_eval-src/shared/integrity.go`. The source file is gitignored.
  Don't rotate the secret casually — it invalidates every existing
  `results.md`.
