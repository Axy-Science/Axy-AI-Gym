# Problem Solving Gym — CLAUDE.md

Astro static-site scoreboard for an internal coding-problem gym. Problems
live on disk under `problems/<slug>/`; each developer commits a solution +
a `results.md` (generated by the problem's `evaluate.sh`); the site reads
all of that at build time and renders ranked leaderboards.

## Repo layout

```
src/
├── layouts/Layout.astro       # global tokens, fonts, dark theme
├── lib/data.ts                # getAllProblems(), getProblemWithScoreboard(slug)
└── pages/
    ├── index.astro            # homepage — problem cards (newest first)
    └── problems/[slug].astro  # dynamic route — markdown + leaderboard

problems/
└── <slug>/
    ├── problem.md             # title/slug/published_at frontmatter + body
    ├── evaluate.sh            # bash <solution_cmd> <output_path> → results.md
    ├── tests/                 # input-N.txt + expected-N.txt pairs
    └── submissions/<handle>/  # solution.* + results.md
```

## Local dev

```
npm install
npm run dev      # http://localhost:4321
npm run build    # dist/
npm run preview  # serve dist/ on :4321
```

## Your first submission in 5 minutes (participant tutorial)

Pick **FizzBuzz Supreme** for this walkthrough — it's `problem-1`.

### 1. Clone and boot the site

```
git clone https://github.com/Axy-Science/Axy-AI-Gym.git
cd Axy-AI-Gym
npm install
npm run dev
```

Open `http://localhost:4321/`. You'll see two problem cards. Click
**FizzBuzz Supreme** — that's the page your row will land on.

### 2. Read the problem

Read `problems/problem-1/problem.md` (or the rendered page — same content).
Note the I/O contract: integers from stdin, one result per line.

### 3. Make your submission folder

Use your GitHub handle as the folder name. Examples here use `aarsh`.

```
mkdir -p problems/problem-1/submissions/aarsh
```

### 4. Write your solution (any language)

The eval script doesn't care about your language — just give it a runnable
command. Quick Python stub at `problems/problem-1/submissions/aarsh/solution.py`:

```python
import sys
for line in sys.stdin:
    n = int(line.strip())
    if n % 15 == 0: print("FizzBuzz")
    elif n % 3 == 0: print("Fizz")
    elif n % 5 == 0: print("Buzz")
    else: print(n)
```

Node, Go, Rust, anything stdin/stdout — all fine.

### 5. Run the eval

From inside the problem folder:

```
cd problems/problem-1
bash evaluate.sh "python3 submissions/aarsh/solution.py" submissions/aarsh/results.md
```

On success you'll see:

```
=============================
  Score: 100 / 100
  Results written to: submissions/aarsh/results.md
=============================
```

Examples for other stacks:

```
bash evaluate.sh "node submissions/aarsh/solution.js" submissions/aarsh/results.md
bash evaluate.sh "go run submissions/aarsh/solution.go" submissions/aarsh/results.md
```

### 6. Iterate

If a test fails, the generated `results.md` shows expected vs. actual for
each failing case. Read it, fix the bug, re-run the eval — it overwrites
the previous `results.md` with a fresh timestamp.

The site auto-refreshes in dev mode whenever `results.md` changes, so you
can watch your row update live at `http://localhost:4321/problems/problem-1`.

### 7. Commit and open a PR

```
git checkout -b aarsh/problem-1
git add problems/problem-1/submissions/aarsh
git commit -m "solve: problem-1 (aarsh, 100/100)"
git push origin aarsh/problem-1
gh pr create --fill
```

Once your PR merges to `main`, the deployed site rebuilds and your row
appears on the public scoreboard.

### Scoring & tiebreakers

- Ranks sort by **score descending**.
- Ties break by **submission timestamp ascending** — earliest wins. So if
  two people both hit 100/100, the one who got there first ranks higher.

## Adding a new problem (maintainers)

1. `mkdir -p problems/<slug>/{tests,submissions}`
2. Write `problem.md` with frontmatter:
   ```
   ---
   title: "Problem Title"
   slug: "<slug>"               # must match folder name
   published_at: YYYY-MM-DD
   ---
   ```
3. Drop input/expected pairs in `tests/`: `input-1.txt`, `expected-1.txt`, …
4. Author `evaluate.sh`. Required interface: `<solution_cmd> <output_path>` →
   writes `results.md` whose frontmatter is exactly:
   ```
   ---
   author: "<handle>"
   total_points: <int>
   score: <int>
   submitted_at: "<ISO 8601 UTC>"
   ---
   ```
5. `chmod +x evaluate.sh`
6. Test it with a known-good solution before committing.

When writing eval scripts, **don't extract the test number with
`grep -o '[0-9]\+' "$full_path"`** — it matches digits from the whole
path (including `problem-1`) and breaks lookup. Use basename slicing:

```
base="${input_file##*/}"; test_num="${base#input-}"; test_num="${test_num%.txt}"
```

…or an explicit `for i in 1 2 3` loop when the test count is fixed.

## Notes for AI assistants working in this repo

- **Data is filesystem-driven.** `src/lib/data.ts` reads `problems/` at build
  time via `process.cwd()` + `fast-glob`. No DB, no API. Touching files
  invalidates the dev build automatically.
- **Markdown rendering:** problem body → `marked.parse()` → `<article set:html>`.
  Markdown's first `<h1>` is hidden via CSS since the page already shows a
  styled title.
- **Dynamic routes:** `[slug].astro` needs `getStaticPaths()` returning every
  problem slug; without it Astro can't pre-render those pages.
- **Design tokens:** all colors, fonts, spacing live as CSS variables on
  `:root` inside `src/layouts/Layout.astro`. Edit there, not per-page.
- **Fonts:** `Press Start 2P` is marquee-only (unreadable at body sizes);
  `JetBrains Mono` is everything else.
- **Ranking tiebreaker** is dense — entries with identical `(score,
  submitted_at)` share a rank. Implemented in `getProblemWithScoreboard`.
- **macOS case quirk:** the filesystem is case-insensitive but git is
  case-sensitive. Don't reintroduce a lowercase `readme.md` alongside
  `README.md`.
